# GitHub Trending 候选筛选准则

选择 3 到 5 个项目时，优先保留高信号项目，不追求覆盖全部热门仓库。

## 优先信号

- 新的工程工作流已经形成明显讨论热度
- 项目反映出开发者工具、AI、自动化、数据基础设施、前端体验的方向变化
- 仓库的使用方式足够清晰，能推导出对工程团队的实际价值
- 项目不是一次性话题，而是可能形成后续跟进价值

## 降权信号

- 纯娱乐、梗图、玩具性质项目
- 与更强候选表达相同趋势的重复项目
- 页面信息太少，无法负责地说明项目价值
- 仅因为争议或短时传播而上榜，但缺少技术含量

## 判断口径

- 把“热门”与“重要”分开。Trending 只说明当前关注度，不自动等于工程价值。
- 如果无法确认具体原因，用“可能因为……”而不是“是因为……”。
- 如果项目很热但难以落地，也可以保留，但动作建议通常应为 `继续观察` 而不是 `立即试用`。
- 优先给出可执行判断，而不是罗列仓库介绍。
